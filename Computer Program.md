## 一、問題解決與運算思維

### 1. 什麼是運算思維？

**運算思維（Computational Thinking）** 不是叫同學「只能寫程式」才能實現運算思維，而是一種**用電腦可以理解的方式來解決問題的思考方法**。即使你從未寫過一行程式碼，運算思維也能幫助你更有系統地分析複雜問題、設計解決方案。

> 當然，如果你想要實現運算思維的思考內容，通常最簡單的方式就是實作程式。

運算思維由四個核心能力組成，它們互相支撐，共同構成解決問題的完整框架。

### 2. 問題分解（Decomposition）

面對一個複雜的問題，第一步不是立刻尋找答案，而是將它**拆解成較小、較易處理的子問題**。每個子問題可以獨立解決，最後再將答案組合起來。

這個概念在工程領域極為普遍。蓋一棟大樓不是一個問題，而是地基、結構、水電、裝潢……數十個子問題的組合。同樣地，撰寫一套工廠管理系統，也可以拆解為訂單管理、庫存追蹤、品質紀錄、報表產生等獨立模組。

**工業應用：生產流程拆解**

一條完整的生產線看似複雜，但透過問題分解，可以將其拆為：原料入庫 → 備料 → 加工站A → 加工站B → 品質檢測 → 包裝 → 出貨。每個步驟都有明確的輸入（上一站的半成品）與輸出（下一站的輸入），這種結構化的拆解方式，正是撰寫生產管理程式的基礎。

### 3. 模式辨識（Pattern Recognition）

解決一個新問題時，我們往往可以發現它與過去解決過的問題有相似之處。**模式辨識**是找出問題中重複出現的規律或結構，藉此套用已知的解法，而不必每次都從零開始。

例如，「計算每條生產線的日平均產量」與「計算每位業務員的月平均銷售額」在結構上完全相同：都是「對一組數字取平均值」。辨識出這個共通模式，同一段程式邏輯就能處理兩種情境。

**工業應用：作業標準化（SOP）**

工廠的標準作業程序（SOP）正是模式辨識的體現：不同的機台、不同的零件，若其加工流程遵循相同的「夾緊 → 加工 → 量測 → 記錄」模式，就可以用同一套程式框架來管理，只需替換各步驟的參數即可。

### 4. 抽象化（Abstraction）

**抽象化**是指忽略問題中不重要的細節，只保留對解決問題真正有用的資訊。這讓我們能夠專注在本質上，而不被枝微末節淹沒。

一個好例子是地圖：真實的城市遠比地圖複雜，但地圖省略了建築物的顏色、路旁的樹木，只保留道路、地名等關鍵資訊，反而讓導航變得可行。程式中的「函數」（後面會介紹）正是抽象化的實踐——你不需要知道 `計算平均值()` 這個函數內部如何運作，只需要知道「給它一串數字，它會回傳平均值」。

### 5. 演算法思考（Algorithmic Thinking）

**演算法（Algorithm）** 是解決特定問題的**一系列明確、有限、有序的步驟**。演算法思考是將問題的解法，轉化為可以讓電腦逐步執行的清晰指令序列。

演算法必須具備幾個特性：

- **明確性**：每一個步驟必須清楚無歧義，不能有「大概做一下」這種模糊指令
- **有限性**：必須在有限步驟內結束，不能無止境地執行
- **有效性**：每一個步驟都是電腦可以實際執行的操作
- **輸入與輸出**：有明確的輸入資料，並產生預期的輸出結果

**工業應用：標準作業流程（SOP）與演算法的相似性**

工廠的 SOP 與演算法在本質上非常相近：兩者都是「明確的步驟序列」，都有輸入（原料、條件）與輸出（成品、結果），都強調每個步驟的清晰性。主要差異在於，SOP 的執行者是人，而演算法的執行者是電腦。將 SOP 轉化為演算法，正是「流程數位化」的第一步。

> **小結**：運算思維的四個面向——分解、模式辨識、抽象化、演算法思考——不是孤立的技巧，而是解決複雜問題時自然流動的思考過程。在面對工廠流程改善、系統設計或資料分析任務時，有意識地運用這四個思維工具，往往能讓問題變得清晰可解。

## 二、程式設計基本概念

> 系上的「程式設計」課程會把這一節的概念講得更清楚，這裡你可以建立最基本的概念。

### 1. 程式的本質

一個程式，無論多複雜，其本質都可以用三個字描述：**輸入 → 處理 → 輸出**。

```
輸入（Input）  →  處理（Process）  →  輸出（Output）
  原始資料          運算與邏輯          結果與資訊
```

- **輸入**：程式接收的資料，來源可以是鍵盤輸入、檔案讀取、感測器訊號、資料庫查詢……
- **處理**：程式對資料進行的計算、判斷、轉換等操作
- **輸出**：處理完成後產生的結果，可以是螢幕顯示、檔案寫入、發送指令給機台……

**工業應用範例：良率計算程式**

- 輸入：今日生產總數（500件）、不良品數（12件）
- 處理：良率 = (500 - 12) / 500 × 100%
- 輸出：良率 97.6%，顯示於報表並存入資料庫

這個例子雖然簡單，但結構與任何複雜程式完全相同。

### 2. 變數與資料型態

**變數（Variable）** 是程式中用來儲存資料的「容器」，有名稱（讓你找得到它）和值（實際儲存的內容）。就像倉庫中貼了標籤的儲位：「A-01」儲位裡放著「齒輪，數量：50」。

不同性質的資料需要不同的**資料型態（Data Type）** 來儲存，這影響了資料在記憶體中的表示方式與可進行的操作：

| 資料型態 | 說明 | 工業範例 |
| --- | --- | --- |
| **整數（Integer）** | 沒有小數的數字 | 產品數量、批次編號、機台編號 |
| **浮點數（Float）** | 帶有小數的數字 | 溫度 23.5°C、良率 0.976、重量 1.25 kg |
| **字串（String）** | 文字資料，以引號括住 | 產品名稱、操作員姓名、機台型號 |
| **布林（Boolean）** | 只有兩種值：真（True）或假（False） | 機台是否運作中、零件是否合格 |

**為什麼資料型態重要？**

資料型態錯誤是程式常見的錯誤來源。例如，若將產品數量儲存為字串「50」而非整數 50，程式在計算「50 + 30」時可能得到字串「5030」而非數字 80。在工業系統中，這類型別錯誤可能導致產量統計錯誤，造成錯誤的生產決策。

### 3. 基本運算

程式可以對資料進行兩大類運算：

**算術運算（Arithmetic Operations）**：

| 運算 | 符號 | 範例 | 結果 |
| --- | --- | --- | --- |
| 加法 | `+` | `良品數 = 總數 - 不良數` | 488 |
| 減法 | `-` | `500 - 12` | 488 |
| 乘法 | `*` | `單價 * 數量` | 總金額 |
| 除法 | `/` | `488 / 500` | 0.976 |
| 取餘數 | `%` | `17 % 5` | 2（17除以5餘2） |

**邏輯運算（Logical Operations）**：

邏輯運算用於判斷條件，結果為真（True）或假（False）：

| 運算 | 說明 | 工業範例 |
| --- | --- | --- |
| `AND`（且） | 兩個條件都為真，結果才為真 | 溫度正常 AND 壓力正常 → 製程正常 |
| `OR`（或） | 任一條件為真，結果即為真 | 溫度異常 OR 壓力異常 → 觸發警報 |
| `NOT`（非） | 將真假反轉 | NOT 合格 → 不合格 |

**工業應用：生產數據計算**

```
今日產量    = 500
不良品數    = 12
良品數      = 今日產量 - 不良品數    → 488
良率        = 良品數 / 今日產量      → 0.976
良率百分比  = 良率 * 100             → 97.6%

是否達標    = 良率 >= 0.95           → True（達標）
```

## 三、控制結構

### 1. 為什麼需要控制結構？

若程式只能從頭到尾執行一遍，它能做的事情非常有限。**控制結構（Control Structure）** 讓程式能夠根據條件做出不同的選擇，或重複執行某些步驟，使程式具備真正的「邏輯判斷」與「自動化」能力。

### 2. 條件判斷（if-else）

**條件判斷**讓程式根據某個條件是否成立，選擇執行不同的程式碼區塊。其基本邏輯是：

```
如果 [條件成立]:
    執行 A
否則:
    執行 B
```

可以用流程圖來理解：

```
          ┌─────────────┐
          │  條件判斷？  │
          └──────┬──────┘
         成立    │    不成立
        ┌────────┴────────┐
        ↓                 ↓
   執行 A 區塊        執行 B 區塊
        └────────┬────────┘
                 ↓
            繼續往下執行
```

**工業應用：品質檢測條件判斷**

```
量測值 = 讀取感測器()

如果 量測值 >= 下限 AND 量測值 <= 上限:
    判定 = "合格"
    記錄至良品區
否則如果 量測值 < 下限:
    判定 = "過小，不合格"
    記錄至不良品區，觸發警報
否則:
    判定 = "過大，不合格"
    記錄至不良品區，觸發警報

列印報告(量測值, 判定)
```

這個邏輯可以嵌入 PLC 程式或 MES 系統，讓機器自動分類產品，無需人工逐一判斷。

### 3. 迴圈（for、while）

**迴圈（Loop）** 讓程式能夠重複執行一段程式碼，直到某個條件達成。這正是電腦「不知疲倦」的關鍵——它可以用同樣的速度重複執行一個操作一萬次，而不會因疲勞而出錯。

**for 迴圈**：適合用在**已知重複次數**的情境。

```
對 工單清單 中的每一張工單:
    讀取工單內容
    計算所需原料
    更新庫存
    列印領料單
```

**while 迴圈**：適合用在**重複到某條件成立為止**的情境。

```
當 生產線尚未完成今日配額:
    繼續生產
    更新已生產數量
    如果原料不足: 發出補料警報
```

**工業應用：批次作業**

工廠常見的批次作業（Batch Processing）天然適合用迴圈表達：

```
批次清單 = [批次001, 批次002, 批次003, ..., 批次050]

對 批次清單 中的每一個批次:
    投入原料
    執行加工流程
    進行品質量測
    記錄批次結果
    如果良率 < 95%: 發出警告，等候工程師確認
```

### 4. 流程控制（break 與 continue 概念）

在迴圈執行過程中，有時需要提前結束或跳過某次執行：

- **break（中斷）**：立即結束整個迴圈，跳出迴圈繼續往下執行。常用於找到目標後不需繼續搜尋，或偵測到嚴重異常需立即停止的情境。
- **continue（繼續）**：跳過本次迴圈剩餘的步驟，直接進入下一次迴圈。常用於略過不需處理的特殊情況。

**工業情境範例：**

```
對 感測器資料清單 中的每一筆資料:
    如果 資料 是空值（感測器離線）:
        continue    ← 跳過這筆，繼續下一筆
    如果 溫度 > 緊急上限:
        觸發緊急停機
        break       ← 立即停止所有處理，進入緊急程序
    正常分析資料
```

> **小結**：條件判斷讓程式「有邏輯」，迴圈讓程式「有效率」。這兩種控制結構的組合，是幾乎所有自動化程式的核心骨架。在工業場域中，大量重複性的判斷與計算工作——從感測器資料篩選到批次報表生成——都可以透過這兩種結構自動化完成。

## 四、函數與模組化

### 1. 函數的概念

**函數（Function）** 是一段被命名、可重複呼叫的程式碼區塊，用來完成一個特定的任務。函數接受**輸入（參數，Parameter）**，執行處理，並回傳**輸出（回傳值，Return Value）**。

用一個工廠機台的比喻來理解：一台**計算良率的機器**，你放入「總產量」和「不良品數」（輸入），它自動計算後輸出「良率百分比」（輸出）。每次需要計算良率時，直接使用這台機器，不需重新製造一台。

```
函數定義：
    計算良率(總產量, 不良品數):
        良品數 = 總產量 - 不良品數
        良率   = 良品數 / 總產量 * 100
        回傳   良率

函數呼叫：
    A線良率 = 計算良率(500, 12)    → 回傳 97.6
    B線良率 = 計算良率(320,  8)    → 回傳 97.5
    C線良率 = 計算良率(410, 31)    → 回傳 92.4
```

同樣的計算邏輯只寫一次，卻可以在任何需要的地方重複使用。

### 2. 重複使用與模組化設計

**模組化（Modularization）** 是將程式切分為多個功能獨立的函數（或模組），每個函數只負責一件事，並且可以被其他程式重複呼叫。這帶來幾個重要優點：

- **可讀性**：程式邏輯清晰，每個函數名稱就說明了它的用途
- **可維護性**：修改一個函數不影響其他部分；若計算方式改變，只需修改一處
- **可重用性**：同一個函數可被不同的程式或模組呼叫，避免重複撰寫相同邏輯
- **可測試性**：每個函數可以獨立測試，確保其正確性後再整合

**工業應用：將生產流程拆成可重複的作業單元**

一套完整的生產日報系統，可以用模組化的方式設計：

```
函數：讀取感測器資料(機台編號, 日期)
函數：計算良率(總產量, 不良品數)
函數：判斷是否達標(良率, 目標良率)
函數：生成日報表(機台清單, 日期)
函數：寄送警報郵件(收件人, 警報內容)

主程式：
    對 每台機台:
        資料   = 讀取感測器資料(機台, 今天)
        良率   = 計算良率(資料.產量, 資料.不良數)
        達標   = 判斷是否達標(良率, 95%)
        如果 NOT 達標:
            寄送警報郵件(工程師, "機台X良率不達標")
    生成日報表(所有機台, 今天)
```

每個函數職責單一、邊界清晰，整個系統的邏輯一目了然。

## 五、基本資料處理

### 1. 陣列與列表（List）

到目前為止，我們討論的變數一次只儲存一個值。但實際工作中，我們往往需要處理**一組同類型的資料**，例如：今天 24 小時內每小時的溫度量測值、本週五天的每日產量。

**陣列（Array）** 或 **列表（List）** 是一種資料結構，可以在一個變數名稱下，依序儲存多個值，並透過**索引（Index）** 存取特定位置的值。

```
每日產量 = [480, 512, 498, 467, 521]
             ↑     ↑     ↑     ↑     ↑
           [0]   [1]   [2]   [3]   [4]   ← 索引從 0 開始

每日產量[0] → 480（星期一）
每日產量[2] → 498（星期三）
每日產量[4] → 521（星期五）
```

> **注意**：大多數程式語言的索引從 0 開始，而非從 1 開始。這是初學者最常見的混淆點之一，請特別留意。

有了列表，就能用迴圈輕鬆處理一整週的資料：

```
每日產量 = [480, 512, 498, 467, 521]
總產量   = 0

對 每日產量 中的每一個值:
    總產量 = 總產量 + 該值

平均產量 = 總產量 / 5    → 495.6
```

**工業應用：生產紀錄與庫存清單**

列表在工業資訊系統中無所不在：

- `感測器讀值 = [23.1, 23.4, 23.2, 24.8, 23.0, ...]`（每分鐘溫度）
- `工單清單 = ["WO-001", "WO-002", "WO-003", ...]`（待加工工單）
- `不良原因 = ["尺寸超差", "表面刮傷", "尺寸超差", "缺料", ...]`（品質記錄）

### 2. 簡單資料讀取與整理

在工業環境中，資料往往來自外部檔案（如 CSV 格式的感測器匯出資料）或資料庫查詢。基本的資料處理流程包含以下幾個步驟：

**1. 讀取資料**：從檔案、資料庫或感測器取得原始數據，存入列表或其他資料結構。

**2. 篩選（Filtering）**：從大量資料中，只取出符合條件的資料。例如，從一個月的感測器紀錄中，篩選出溫度超過上限的所有時間點。

**3. 排序（Sorting）**：將資料按照特定順序排列，例如將工單依優先級由高到低排列，或將不良品記錄依時間排序。

**4. 彙整（Aggregation）**：計算統計值，如總計、平均、最大值、最小值、計數等，將大量資料壓縮為有意義的摘要資訊。

**5. 輸出（Output）**：將處理後的結果寫入報表、存回資料庫，或顯示於監控畫面。

```
原始資料（從感測器 CSV 讀入）：
溫度紀錄 = [23.1, 24.5, 23.8, 26.2, 23.4, 25.9, 27.1, 23.0]

篩選異常（溫度 > 25.0）：
異常紀錄 = [26.2, 25.9, 27.1]

彙整：
最高溫度 = 最大值(溫度紀錄)   → 27.1
平均溫度 = 平均值(溫度紀錄)   → 24.6
異常次數 = 長度(異常紀錄)     → 3
```

## 第二篇：資料結構基礎

---

## 六、資料結構的角色

### 1. 為何需要資料結構？

想像一間沒有任何整理系統的倉庫：所有零件散落各地，每次要找某個零件都需要翻遍整個倉庫。這種倉庫雖然能儲存零件，但效率極差。現在想像同一間倉庫，依照零件類型分區、每個儲位都有編號、入庫出庫都有記錄——找東西不再是問題。

**資料結構（Data Structure）** 就是程式中的「倉庫整理系統」：它決定了資料如何被組織、儲存，以及如何被存取。選擇正確的資料結構，往往是程式效能好壞的關鍵。

同樣是「儲存一批工單」，放進不同的資料結構，存取的效率與適用的操作就大不相同：放進**列表**可以依序處理；放進**佇列**可以確保先到先服務；放進**映射**可以用工單號碼瞬間查到工單內容。

### 2. 抽象資料型態（ADT）

**抽象資料型態（Abstract Data Type，ADT）** 是從「使用者角度」定義資料結構：它**能做什麼操作**，而不涉及內部如何實作。就像你使用電梯，只需要知道「按樓層按鈕，門會打開，電梯會移動到指定樓層」，不需要知道纜繩、馬達的運作細節。

這個概念讓程式設計者能夠專注在「我需要什麼功能」，而不是「我要如何在記憶體中排列資料」，大幅降低了設計的複雜度。

**工業應用：訂單、庫存、工單管理**

| 管理場景 | 需要的操作 | 適合的資料結構 |
| --- | --- | --- |
| 訂單等候處理 | 先進先出、加入尾端、取出頭部 | 佇列（Queue） |
| 操作歷史紀錄（可復原） | 後進先出、加入頂端、取出頂端 | 堆疊（Stack） |
| 用工單號碼查詢工單 | 快速依鍵值查找 | 映射（Map） |
| 依優先序處理工單 | 取出最高優先的項目 | 優先佇列 |
| 組織物料清單層級 | 上下層關係、子節點存取 | 樹（Tree） |

## 七、陣列與列表（Array / List）

### 1. 順序儲存概念

陣列與列表是最基本、最直觀的資料結構：將多個資料元素**依照順序排列在一起**，每個元素有一個對應的**索引（Index）** 指示其位置。

在記憶體中，陣列的元素通常是**連續排列**的，就像超市貨架上一排緊鄰的商品——你知道第一個的位置，就能直接計算第 n 個在哪裡，不需逐一搜尋。這使得**依索引存取**（如「給我第3個元素」）的速度極快，無論陣列有多長，都能在固定時間內完成。

```
記憶體中的陣列示意：
位置:  [100] [101] [102] [103] [104]
內容:   480   512   498   467   521
索引:   [0]   [1]   [2]   [3]   [4]
```

### 2. 存取與遍歷

對陣列最常見的兩種操作：

**隨機存取（Random Access）**：直接透過索引取得特定位置的元素。速度很快，不論陣列多長，存取任意一個元素的時間相同。

```
每日產量[3] → 直接取得第4天的產量 467
```

**遍歷（Traversal）**：從頭到尾依序訪問陣列中的每一個元素，通常搭配迴圈使用。

```
對 索引 從 0 到 4:
    印出 "第" + (索引+1) + "天產量：" + 每日產量[索引]

輸出：
第1天產量：480
第2天產量：512
...
```

### 3. 工業應用：每日產量紀錄

**每日產量記錄**：將一個月 30 天的每日產量存入列表，可以輕鬆計算月總量、月平均量、找出最高產日與最低產日：

```
月產量紀錄 = [480, 512, 498, 467, 521, 489, 503, ...]  ← 共30個值

月總量   = 加總(月產量紀錄)
月平均   = 月總量 / 30
最高產日 = 找最大值的索引(月產量紀錄) + 1
最低產日 = 找最小值的索引(月產量紀錄) + 1
```

**感測器時序資料**：每 10 秒記錄一次溫度，一天下來有 8,640 筆資料，存入列表後，可以用迴圈掃描找出異常時段，或計算各時段的平均溫度。

## 八、堆疊與佇列（Stack & Queue）

### 1. 兩種截然不同的存取方式

堆疊與佇列都是管理「一序列資料」的結構，但它們規定資料進出的順序截然不同，也因此適合不同的應用場合。

### 2. 堆疊（Stack）——後進先出（LIFO）

**堆疊（Stack）** 遵循 **LIFO（Last In, First Out，後進先出）** 的原則：最後放入的資料，最先被取出。

想像一疊盤子：你只能從最頂端放入（push）或取出（pop）盤子，不能從中間抽取。

```
堆疊操作示意：
            Push(C)     Pop()
   [C] ←─────────────────────→ [C] 被取出
   [B]                           [B]
   [A]                           [A]
（頂端）                        （頂端）
```

堆疊的兩個基本操作：
- **Push（推入）**：將新元素放到堆疊頂端
- **Pop（彈出）**：取出並移除堆疊頂端的元素

**工業應用：流程回溯與操作步驟檢查**

堆疊最常見的應用是「復原（Undo）」功能：每次操作都被推入堆疊，按下「復原」時，從頂端彈出最後一次操作並執行反向動作。

在工業環境中，**設備操作步驟的檢查**也適合用堆疊模型：某些設備的啟動程序必須嚴格按照順序執行，關閉時必須以相反順序（後啟動的先關閉）拆除，確保安全。

```
啟動順序（Push）：
  步驟1：啟動冷卻系統   → Stack: [冷卻系統]
  步驟2：啟動液壓系統   → Stack: [冷卻系統, 液壓系統]
  步驟3：啟動主軸馬達   → Stack: [冷卻系統, 液壓系統, 主軸馬達]

關閉順序（Pop，後進先出）：
  先關閉 主軸馬達  （最後啟動，最先關閉）
  再關閉 液壓系統
  最後關閉 冷卻系統
```

### 3. 佇列（Queue）——先進先出（FIFO）

**佇列（Queue）** 遵循 **FIFO（First In, First Out，先進先出）** 的原則：最先放入的資料，最先被取出。

這正是日常生活中排隊的邏輯：先到者先被服務，後到者排在後面等待。

```
佇列操作示意：
入隊方向 →  [A][B][C]  → 出隊方向

Enqueue(D)：[A][B][C][D]
Dequeue()： A 被取出，剩 [B][C][D]
```

佇列的兩個基本操作：
- **Enqueue（入隊）**：將新元素加入佇列尾端
- **Dequeue（出隊）**：取出並移除佇列頭部的元素

**工業應用：生產排程與等候加工**

佇列幾乎是所有**排隊等候系統**的天然模型：

- **生產排程**：工單依照下達時間排入佇列，機台空出時，從佇列頭部取出下一張工單開始加工，確保先下達的工單先被處理
- **訂單處理系統**：客戶訂單依照收到時間排入佇列，客服人員依序處理
- **列印工作排隊**：多人同時列印時，列印工作依序排入佇列，印表機一次處理一個

```
生產排程佇列：
初始：[WO-001, WO-002, WO-003, WO-004]

機台完成 WO-001：
  Dequeue → 取出 WO-001 開始加工
  佇列剩：[WO-002, WO-003, WO-004]

新工單到達：
  Enqueue(WO-005)
  佇列：[WO-002, WO-003, WO-004, WO-005]
```

> **小結**：堆疊（後進先出）與佇列（先進先出）分別對應兩種截然不同的現實場景。工廠中的「等候加工隊列」是佇列；「操作步驟的逆序回溯」是堆疊。理解這兩種結構，能幫助你在設計工業資訊系統時，為不同的等候或順序問題選擇正確的資料模型。

## 九、集合與映射（Set / Map）

### 1. 集合（Set）——唯一性

**集合（Set）** 是一種只儲存**不重複元素**的資料結構。當你將資料加入集合時，若相同的值已經存在，集合會自動忽略，不會產生重複。

```
已出現的不良原因 = {}（空集合）

加入 "尺寸超差"  → {"尺寸超差"}
加入 "表面刮傷"  → {"尺寸超差", "表面刮傷"}
加入 "尺寸超差"  → {"尺寸超差", "表面刮傷"}  ← 重複，不加入
加入 "缺料"      → {"尺寸超差", "表面刮傷", "缺料"}
```

集合最常用於：快速判斷某個值**是否已出現**、從大量資料中取得**不重複的種類清單**。

**工業應用範例**：從一個月的不良品紀錄中，快速找出「本月共出現了哪些不良類型」，而不用逐一去除重複值。

### 2. 映射（Map）——鍵值對

**映射（Map）**，又稱字典（Dictionary），是一種儲存**鍵值對（Key-Value Pair）** 的資料結構：每個**鍵（Key）** 對應一個**值（Value）**，就像現實中的字典——查「apple」這個字（鍵），得到它的意思（值）。

```
設備狀態表 = {
    "M-001": "運作中",
    "M-002": "維修中",
    "M-003": "閒置",
    "M-004": "運作中"
}

查詢：設備狀態表["M-002"]  → "維修中"
更新：設備狀態表["M-002"] = "運作中"
新增：設備狀態表["M-005"] = "閒置"
```

映射的核心優勢是：**依鍵查值的速度極快**，不論映射中有多少筆資料，查詢時間幾乎固定。這與必須逐一搜尋的列表相比，在大量資料時有顯著的效能差異。

**工業應用：物料編號查詢與設備對應表**

- **物料主檔**：`{"MAT-001": {"名稱": "齒輪", "單位": "個", "安全庫存": 100}, "MAT-002": {...}, ...}` ——用料號瞬間查到完整物料資訊
- **設備對應表**：`{"機台編號": "負責技術員姓名"}` ——發生異常時立即查出聯絡人
- **產品規格表**：`{"產品料號": {"長度": 50.0, "寬度": 30.0, "公差": 0.05}}` ——加工前查詢規格

## 十、樹與層級結構（概念性介紹）

### 1. 什麼是樹？

前面介紹的列表、堆疊、佇列都是**線性**結構——資料一個接一個地排列。但現實世界中，許多資料天然具有**層級（Hierarchical）**關係，例如公司的組織架構、產品的零件組成，這類資料用**樹（Tree）** 來表示最為自然。

樹由**節點（Node）** 和連接節點的**邊（Edge）** 組成：

```
              總經理
             /      \
        製造部       業務部
       /      \        \
    製造一課  製造二課   業務一課
    /    \
 生產線A  生產線B
```

- 最頂端的節點稱為**根節點（Root）**（如「總經理」）
- 有子節點的節點稱為**父節點（Parent）**
- 沒有子節點的節點稱為**葉節點（Leaf）**（如「生產線A」）
- 從根到任一節點的距離稱為**層級（Level）**

### 2. 工業應用：組織架構與產品結構（BOM）

**產品物料清單（BOM，Bill of Materials）** 是樹狀結構在製造業最重要的應用之一。BOM 描述了一個成品由哪些零組件組成，每個零組件又由哪些更小的零件組成，形成一個層級分明的樹狀結構：

```
成品：自行車
├── 車架總成
│   ├── 主車架
│   └── 前叉
├── 傳動系統
│   ├── 踏板組
│   ├── 鏈條
│   └── 齒輪盤
└── 輪組（×2）
    ├── 輪圈
    ├── 輪胎
    └── 輻條（×32）
```

ERP 系統在計算生產一輛自行車需要多少原材料時，就是沿著這棵 BOM 樹向下展開，逐層計算每個子零件的需求量，這個過程稱為 **MRP 展算（Material Requirements Planning）**。

> **小結**：樹狀結構是表達「整體—部分」或「上層—下層」關係的最自然工具。在工業場域中，組織架構、設備層級（工廠 → 產線 → 機台 → 模組）、產品 BOM、工作分解結構（WBS）都是樹狀結構的體現。理解樹的概念，有助於理解 ERP/MES 系統中許多資料組織的方式。

## 十一、演算法基本概念

### 1. 演算法的定義與特性

**演算法（Algorithm）** 是解決特定問題的**一系列明確、有限、有序的步驟**。這個定義在第一章已經介紹過，這裡我們進一步深入。

一個好的演算法必須具備：

- **明確性（Definiteness）**：每個步驟清楚無歧義，電腦能理解並執行
- **有限性（Finiteness）**：必須在有限步驟後終止，不能無限執行下去
- **輸入（Input）**：有零個或多個明確定義的輸入
- **輸出（Output）**：有一個或多個明確定義的輸出結果
- **有效性（Effectiveness）**：每個步驟都是電腦可以實際執行的基本操作

### 2. 演算法與程式的關係

演算法是**概念層面的解題步驟**，程式則是演算法的**具體實作**。同一個演算法可以用不同的程式語言（Python、Java、C++）來實作；同樣地，同一個問題可能有多種不同的演算法，效率各有高低。

```
問題：在一個列表中，找出最大值

演算法（語言無關）：
  1. 將「目前最大值」設為列表第一個元素
  2. 從第二個元素開始，逐一與「目前最大值」比較
  3. 若當前元素比「目前最大值」還大，則更新「目前最大值」
  4. 重複步驟2–3，直到所有元素都比較完畢
  5. 回傳「目前最大值」
```

**工業應用：標準作業流程（SOP）的演算法本質**

工廠的 SOP 與演算法在結構上高度相似：

| 演算法特性 | SOP 對應 |
| --- | --- |
| 明確性 | 每個操作步驟清楚描述，不允許模糊 |
| 有限性 | 作業有明確的開始與結束 |
| 輸入 | 原料、半成品、設備參數 |
| 輸出 | 成品、品質量測結果、生產記錄 |
| 有效性 | 每個步驟都是操作員可實際執行的動作 |

將 SOP 轉化為電腦可執行的演算法，正是「流程數位化與自動化」的核心工作。

---

## 十二、基本演算法範例

### 1. 搜尋——如何找到你要的資料

**搜尋（Search）** 是在一組資料中，找出符合特定條件的元素。這是程式中最頻繁的操作之一。

**線性搜尋（Linear Search）** 是最直觀的搜尋方式：從頭到尾逐一檢查每個元素，直到找到目標或確認不存在。

```
線性搜尋流程：
目標：找出機台 "M-003" 的位置

機台清單 = ["M-001", "M-002", "M-003", "M-004", "M-005"]

步驟1：檢查 "M-001"，不是目標，繼續
步驟2：檢查 "M-002"，不是目標，繼續
步驟3：檢查 "M-003"，找到目標！→ 回傳索引 2
```

線性搜尋的優點是邏輯簡單，不需要資料事先排序；缺點是若資料量很大，且目標在末端或不存在，需要逐一檢查每一筆資料，效率較低。這正是後面「效率分析」要討論的核心議題。

### 2. 排序——讓資料有序

**排序（Sorting）** 是將一組資料依照特定順序（如數字大小、字母順序、日期先後）重新排列。排序後的資料往往更容易搜尋、分析和呈現。

以**泡沫排序（Bubble Sort）** 為例說明排序的基本概念（直觀但非最有效率）：

```
初始：[64, 34, 25, 12, 22]

第一輪比較（相鄰元素兩兩比較，大的往右移）：
[34, 25, 12, 22, 64]  ← 64 沉到最右

第二輪：
[25, 12, 22, 34, 64]  ← 34 沉到倒數第二

...持續進行直到全部排序完成...

最終：[12, 22, 25, 34, 64]
```

> **重點在流程，不在細節**：本章的目的不是要你記住泡沫排序的每一個步驟，而是理解「排序是透過反覆比較與交換，逐漸將資料調整到正確位置」的這個基本流程。實際工作中，程式語言都已內建高效的排序函數，不需要自己實作。

**工業應用**：

- 將工單依照「截止日期」排序，確保緊急工單優先處理
- 將不良品依照「不良次數由多到少」排序，找出最嚴重的品質問題（柏拉圖分析的基礎）
- 將設備依照「OEE 數值由低到高」排序，找出最需要改善的機台

## 十三、效率與時間複雜度概念

### 1. 為何需要效率分析？

同一個問題，可以用不同的演算法解決，但效率可能天差地別。當資料量小時，差異或許感覺不出來；當資料量增長到成千上萬，甚至數百萬筆時，低效率的演算法可能讓一個本來幾秒鐘就能完成的計算，變成需要等待幾小時。

**工業情境**：你的工廠有 1,000 筆庫存品項，搜尋某個料號可能只要 0.001 秒。但當品項增長到 1,000,000 筆時，同樣的搜尋方法會需要 1 秒甚至更長。若每次領料都需要搜尋一次，系統的回應速度就會明顯變慢。

這就是為什麼電腦科學家要研究演算法效率——不是因為喜歡數學，而是因為**規模增大時，效率的差異會被放大到無法忽視**。

### 2. 時間複雜度（Time Complexity）——直覺理解

**時間複雜度** 描述的是：當輸入資料量（通常以 n 表示）增加時，演算法所需的執行時間如何成長。我們用 **大 O 符號（Big-O Notation）** 來表示，它描述的是「最壞情況下的成長趨勢」。

#### O(1)——常數時間

無論 n 有多大，執行時間**固定不變**。

```
範例：透過索引直接存取列表中的某個元素

每日產量[3] → 無論列表有 5 個還是 5,000 個元素，
              存取第 4 個元素的時間完全相同
```

這是最理想的情況。映射（Map）的查詢操作通常也接近 O(1)。

**工廠類比**：倉庫有固定編號的儲位，只要知道儲位號碼，直接走過去拿，不需要搜尋，不管倉庫有多大，時間一樣。

#### O(n)——線性時間

執行時間**與 n 成正比**：資料量變成 2 倍，時間也大約變成 2 倍。

```
範例：線性搜尋

在 100 筆資料中搜尋 → 最多 100 次比較
在 1,000 筆資料中搜尋 → 最多 1,000 次比較
在 10,000 筆資料中搜尋 → 最多 10,000 次比較
```

**工廠類比**：倉庫沒有分區，要找一個零件只能從頭開始逐一尋找。倉庫裡的零件越多，找到目標平均要花的時間就越長，且成正比。

#### O(n²)——平方時間

執行時間**與 n 的平方成正比**：資料量變成 2 倍，時間變成 4 倍；資料量變成 10 倍，時間變成 100 倍。

```
範例：泡沫排序（對 n 個元素排序，需要進行約 n×n 次比較）

排序 10 筆資料  → 約 100 次比較
排序 100 筆資料 → 約 10,000 次比較
排序 1,000 筆資料 → 約 1,000,000 次比較
```

當 n 不大時，O(n²) 算法勉強可以接受；但一旦 n 變大，效能就會急速惡化。

**直覺式比較**（假設每次基本操作花 1 微秒）：

| n（資料量） | O(1) | O(n) | O(n²) |
| --- | --- | --- | --- |
| 10 | 1 μs | 10 μs | 100 μs |
| 1,000 | 1 μs | 1 ms | 1 秒 |
| 1,000,000 | 1 μs | 1 秒 | **11.6 天** |

這張表清楚地說明了為什麼效率分析很重要——對於百萬筆資料，O(1) 的演算法瞬間完成，O(n) 需要 1 秒，而 O(n²) 需要將近兩週。

```
成長速度視覺化（n = 10 時的比較）：

O(1)  ░ （固定 1 格）
O(n)  ░░░░░░░░░░ （10 格）
O(n²) ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░ （100 格）
```

## 十四、工業情境中的效率思考

### 1. 生產線瓶頸（Bottleneck）類比

演算法效率的概念，與工業工程中的「**瓶頸（Bottleneck）**」概念高度相似。

在一條生產線上，整體的產出速度受限於最慢的那個工站——即使其他工站的效率再高，整體產出仍然無法超過瓶頸工站的速度。改善非瓶頸工站的效率對整體產出幾乎沒有幫助；唯有改善瓶頸，整體才能提升。

同樣地，在一套複雜的程式系統中，整體效能往往由最低效的那個步驟決定。工程師進行程式效能最佳化時，首要任務是找出程式中的「效能瓶頸」（耗時最長的部分），優先改善它，而非平均地對每個部分進行微小優化。

```
生產線效能比喻：

工站A：每分鐘 100 件
工站B：每分鐘  30 件  ← 瓶頸
工站C：每分鐘  80 件

整體產出受限於工站B，每分鐘只能產出 30 件。
提升A到200件/分鐘，整體仍是30件/分鐘。
只有改善B，整體才能提升。

程式效能類比：
步驟1：讀取資料（0.01 秒）
步驟2：資料分析（45 秒）  ← 效能瓶頸
步驟3：輸出報表（0.05 秒）

優先改善步驟2，其餘步驟的優化意義不大。
```

### 2. 排程效率 vs 計算效率

工業工程師熟悉的「排程優化」，在計算效率上是一個著名的困難問題。

**生產排程**的本質，是在眾多工單、機台、工人、截止日期的限制下，找出「最佳」的生產安排。當工單數量少時（如10張），電腦可以窮舉所有可能的排列，找到最佳解。但工單數量增加時，可能的排列數量呈**階乘（n!）成長**——比 O(n²) 還要快得多：

```
10 張工單的排列數：10! = 3,628,800 種
20 張工單的排列數：20! ≈ 2,400,000,000,000,000,000 種（2.4 × 10¹⁸）
```

這就是為什麼實際工廠的排程系統不使用「窮舉所有可能」的方法，而是採用**啟發式演算法（Heuristic Algorithm）**——不保證找到最佳解，但能在合理的時間內找到「夠好的解」。這也是「計算效率」與「排程品質」之間的現實折衷。

### 3. 資料量增加對決策時間的影響

工業4.0帶來了海量的感測器資料，但「資料多」不等於「決策快」——如果資料分析的演算法效率不佳，資料量增加反而可能讓決策系統變慢，甚至失去即時性。

**實際影響場景**：

- **即時品質判斷**：生產線上的視覺檢測系統，每秒需要判斷數十個零件。若判斷演算法的時間複雜度過高，在高速生產線上將無法跟上生產節拍，導致瓶頸。
- **預測性維護**：若每隔 10 分鐘根據最新感測器資料重新訓練預測模型，但模型訓練需要 30 分鐘，系統就永遠趕不上資料的更新速度。
- **供應鏈優化**：當供應商數量、品項數量、工廠數量增加時，優化演算法的複雜度可能以超線性的速度增長，必須採用近似演算法或分散式計算。

> **小結**：效率分析不只是電腦科學家的學術課題，更是工業工程師在設計或評估資訊系統時的重要工具。理解 O(1)、O(n)、O(n²) 的直覺含義，能幫助你在面對「為什麼系統越來越慢」、「增加一倍的機台數量對系統效能有什麼影響」這類問題時，做出更有根據的判斷。

## 十五、資料驅動的流程改善

### 1. 用簡單程式分析生產資料

「資料驅動（Data-Driven）」是工業4.0的核心理念之一：決策不再單純依賴經驗與直覺，而是基於客觀的資料分析。即使是簡單的程式邏輯，也能從生產資料中萃取出有價值的洞見。

一個典型的分析流程：

```
原始資料（一個月的每日良率）：
良率記錄 = [0.976, 0.981, 0.968, 0.943, 0.972, 0.985,
            0.961, 0.978, 0.924, 0.969, ...]

分析步驟：
1. 計算月平均良率：平均(良率記錄)  → 0.962
2. 找出低於目標的日期：篩選(良率 < 0.95)  → [第4天:0.943, 第9天:0.924]
3. 計算低於目標的天數佔比：2/30 → 6.7%
4. 找出最低良率日：最小值(良率記錄) → 第9天 0.924
```

這個分析若每天手動進行，需要幾分鐘；若寫成程式，只需不到 1 秒，且可以在每天生產結束後自動執行，讓工程師能在隔天早上就看到昨天的分析結果。

### 2. 找出異常與趨勢

**異常偵測（Anomaly Detection）** 是資料分析中最直接的應用：設定正常範圍，自動標記超出範圍的資料點。

```
製程溫度監控：
上限 = 25.0°C
下限 = 20.0°C
溫度資料 = [22.1, 23.4, 21.8, 26.2, 22.9, 19.8, 23.1]

異常偵測：
對 每一筆溫度資料:
    如果 溫度 > 上限: 記錄「過高異常」，時間戳，溫度值
    如果 溫度 < 下限: 記錄「過低異常」，時間戳，溫度值

結果：
  過高異常：26.2°C（第4筆）
  過低異常：19.8°C（第6筆）
```

**趨勢分析（Trend Analysis）** 則是觀察資料隨時間的變化方向，例如良率是逐漸下滑還是穩定提升，這往往能在問題爆發前提供預警。

## 十六、自動化思維

### 1. 重複性工作自動化

如果你每天早上都要執行同樣的步驟——打開系統、匯出昨天的資料、貼到 Excel、計算幾個數字、填入報表、寄給主管——這正是程式自動化的最佳目標。

**適合自動化的工作特徵**：

- **重複性高**：每天、每週、每批次都要做一遍
- **規則明確**：步驟固定，判斷標準清楚，不需要複雜的人為判斷
- **資料量大**：人工處理費時費力，且容易出錯
- **時效性要求高**：需要在特定時間內完成（如班次結束後立即產出報表）

### 2. 工業應用：報表自動生成與資料整理

**報表自動生成**是工廠最常見的自動化需求之一：

```
每日生產報表自動化流程：
1. 班次結束（觸發條件）
2. 自動連線 MES 資料庫
3. 查詢今日各機台的產量、良率、稼動率
4. 計算各項 KPI（OEE、達成率、不良率...）
5. 將結果填入標準報表模板
6. 儲存為 PDF 與 Excel 格式
7. 自動寄送給相關人員（班長、工程師、主管）

以往：班長花 30–60 分鐘手動整理
自動化後：程式 5 秒完成，班長可以專注在異常處理
```

**資料整理自動化**：

- 自動將不同機台匯出的 CSV 格式統一化（欄位名稱、日期格式、單位）
- 自動清除明顯的異常值（如感測器離線時回傳的 -999）
- 自動將資料依批次號、日期、機台分類存檔

## 十七、工業4.0概念導入

### 1. 資料 → 資訊 → 決策

工業4.0的核心價值鏈，可以用三個層次來描述：

```
感測器原始資料
（溫度: 23.5, 壓力: 101.2, 振動: 0.03...）
         ↓  [資料處理與分析程式]
有意義的資訊
（主軸溫度正常、液壓系統壓力偏低、振動值在正常範圍內）
         ↓  [決策邏輯]
可執行的決策
（發出液壓系統預警，通知技術員於本班次結束前檢查）
```

程式扮演的正是「資料」到「資訊」這個轉化過程的核心角色：透過計算、比較、判斷、統計，將原始的數字轉化為人類能理解且能採取行動的資訊。

### 2. IoT 資料如何進入程式處理流程

在工業物聯網（IIoT）架構中，感測器資料從產生到被程式利用，通常經過以下流程：

```
[感測器]
  ↓ 類比訊號（溫度、壓力、振動...）
[ADC 轉換]
  ↓ 數位訊號
[PLC / 邊緣裝置]
  ↓ 工業通訊協定（Modbus、OPC UA...）
[資料擷取程式]
  ↓ 數值資料（Float、Integer...）
[資料處理程式]
  ├─ 存入資料庫（歷史紀錄）
  ├─ 即時異常偵測（條件判斷）
  └─ 更新監控儀表板（輸出）
```

每個箭頭之間，都有程式在運作——擷取、轉換、驗證、儲存、分析。理解這個流程，有助於你在面對工業資訊系統的設計或問題排查時，能夠定位問題發生在哪個環節。

### 3. 簡單案例：感測器數據分析流程

以**設備振動監測**為例，展示一個完整的端對端資料處理流程：

```
目標：監測主軸振動，預警異常

資料來源：振動感測器，每秒回傳一個加速度值（m/s²）

程式邏輯：

振動閾值 = 0.5  （正常範圍上限）
警告閾值 = 0.8  （預警門檻）
緊急閾值 = 1.2  （緊急停機門檻）
連續異常計數 = 0

無限迴圈（持續監測）:
    振動值 = 讀取感測器()
    記錄至資料庫(時間戳, 振動值)

    如果 振動值 > 緊急閾值:
        執行緊急停機程序()
        發送緊急警報(所有人員)
        break  ← 退出監測迴圈，等待人員處置

    否則如果 振動值 > 警告閾值:
        連續異常計數 += 1
        如果 連續異常計數 >= 5:  ← 連續5秒超過警告值
            發送預警通知(技術員)
            連續異常計數 = 0  ← 重置計數

    否則:
        連續異常計數 = 0  ← 恢復正常，重置計數

    等待 1 秒
```

## 十八、系統觀點

### 1. 程式 + 資料結構 + 硬體/系統整合

在真實的工業環境中，程式不是孤立運作的，而是嵌入在一個由硬體、網路、作業系統、其他軟體共同構成的複雜系統中。

一個完整的工業資訊系統，從底層到上層可以分為：

```
應用層：ERP、MES、SCADA、BI 儀表板
   ↑ 程式邏輯（演算法 + 資料結構）
平台層：資料庫、訊息佇列、API 服務
   ↑ 作業系統（資源管理、行程調度）
基礎設施層：伺服器、工業電腦、PLC、感測器
   ↑ 硬體（CPU、記憶體、儲存、網路）
```

每一層都依賴下層提供的服務，也為上層提供基礎。程式設計發生在「應用層」與「平台層」，但其效能直接受到下層硬體的限制——這就是為什麼理解硬體（CPU、記憶體、儲存）對程式效能的影響，對工業工程師同樣重要。

### 2. MES / ERP 背後的資料處理概念

ERP 與 MES 這些大型工業軟體，看起來功能複雜，但其背後的資料處理邏輯，往往可以用本教材介紹的基礎概念來理解：

**ERP 的物料需求規劃（MRP）**：

```
本質：BOM 樹的遍歷（樹狀結構）+ 庫存查詢（映射）+ 計算

流程：
  對 每一張生產工單:                 ← 迴圈
    展開 BOM 樹，取得所有子零件      ← 樹的遍歷
    對 每一個子零件:                 ← 迴圈
      查詢現有庫存量                 ← 映射查詢 O(1)
      計算需求量 = 需求 - 庫存       ← 算術運算
      如果 需求量 > 0:               ← 條件判斷
        產生採購申請或生產工單
```

**MES 的生產排程**：

```
本質：工單列表（列表）+ 優先序排列（排序）+ 機台分配（映射）

流程：
  工單佇列 = 讀取所有待排程工單     ← 建立佇列
  依優先級排序工單                  ← 排序演算法
  對 每台可用機台:                  ← 迴圈
    取出優先級最高的工單            ← 佇列出隊
    分配工單到機台                  ← 映射更新
    計算預計完工時間                ← 算術運算
```

這些系統雖然龐大，但其核心邏輯都是由迴圈、條件判斷、資料結構操作所構成的。理解這些基礎，有助於你在使用這些系統時，理解其行為背後的邏輯，也有助於你在與 IT 部門溝通需求時，能夠更精確地表達你的想法。

## 十九、基礎練習

### 1. 數據計算

以下是幾個適合初學者的基礎計算練習，建議用任何你熟悉的工具（Excel 公式、Python、或甚至純粹的邏輯描述）來實作：

**練習1：良率計算**

```
輸入：今日各機台的總產量與不良品數
處理：計算每台機台的良率、全廠平均良率
輸出：各機台良率列表，標示低於目標（95%）的機台
```

**練習2：感測器統計**

```
輸入：一天內每小時的溫度記錄（24筆）
處理：計算最高溫、最低溫、平均溫、超出正常範圍的次數
輸出：統計報告，包含是否有異常的判斷
```

**練習3：庫存警示**

```
輸入：各品項的現有庫存量與安全庫存量
處理：比較現有庫存與安全庫存，找出需要補貨的品項
輸出：需要補貨的品項清單，依缺口大小排序
```

### 2. 條件判斷（品質分類）

**練習：尺寸分類系統**

```
規格：長度目標值 50.0 mm，公差 ±0.1 mm
  → 合格範圍：49.9 mm ~ 50.1 mm
  → 過小：< 49.9 mm
  → 過大：> 50.1 mm

量測值清單 = [50.05, 49.85, 50.12, 50.01, 49.92, 50.15, 49.99]

對 每一個量測值:
    如果 49.9 <= 量測值 <= 50.1:
        分類 = "合格"
    否則如果 量測值 < 49.9:
        分類 = "過小，報廢/重工"
    否則:
        分類 = "過大，報廢/重工"
    記錄(量測值, 分類)

計算：合格率、各類別數量
```

## 二十、小型應用案例

### 1. 簡單庫存管理模擬

設計一個簡單的庫存管理邏輯，支援以下操作：

```
庫存資料（映射）：
{
  "MAT-001": {"名稱": "螺栓M5", "庫存": 500, "安全庫存": 100},
  "MAT-002": {"名稱": "軸承6205", "庫存": 45, "安全庫存": 50},
  "MAT-003": {"名稱": "密封圈", "庫存": 230, "安全庫存": 80}
}

操作1：入庫
  輸入：料號、數量
  處理：庫存[料號] += 數量
  輸出：更新後庫存量

操作2：出庫
  輸入：料號、數量
  處理：如果庫存充足，庫存[料號] -= 數量；否則回報庫存不足
  輸出：成功或庫存不足警告

操作3：庫存盤點報告
  處理：遍歷所有品項，找出庫存低於安全庫存的品項
  輸出：需補貨清單，包含料號、名稱、現有庫存、安全庫存、缺口
```

### 2. 生產排程（簡化版 Queue 模型）

```
情境：一台機台，多張工單等候加工，以 FIFO 原則排程

工單佇列（Queue）：
初始：[WO-001(急), WO-002(一般), WO-003(一般), WO-004(急)]

排程流程：
1. 將工單依優先級（急件優先）與下達時間排入佇列
2. 機台完成一張工單後，自動從佇列頭部取出下一張
3. 若有新工單插入，根據優先級插入佇列適當位置
4. 顯示預估每張工單的開始與完成時間

輸出範例：
WO-001：09:00 開始，10:30 完成（加工時間 90分鐘）
WO-004：10:30 開始，11:30 完成（加工時間 60分鐘）← 急件插隊
WO-002：11:30 開始，12:30 完成
WO-003：12:30 開始，14:00 完成
```

### 3. 生產資料統計報表

```
輸入：一週（5天）各班次（3班）的生產記錄
  格式：日期、班次、機台、產量、不良數、停機時間（分鐘）

處理：
1. 計算每天各班次的良率
2. 計算每台機台的週累計產量
3. 計算各機台的稼動率（實際運作時間 / 計畫運作時間）
4. 找出本週最佳班次（良率最高）與最差班次
5. 列出停機時間超過 30 分鐘的事件

輸出：結構化報表，包含各項 KPI 匯總與異常清單
```

## 二十一、視覺化與呈現（選擇性）

### 1. 基本圖表概念

資料分析的最後一步，往往是將結果視覺化——轉換為圖表，讓人更容易看出趨勢、比較差異、發現異常。不同類型的資料適合不同類型的圖表：

| 圖表類型 | 適合呈現 | 工業應用範例 |
| --- | --- | --- |
| **折線圖（Line Chart）** | 隨時間變化的趨勢 | 每日良率趨勢、每月產量走勢 |
| **柱狀圖（Bar Chart）** | 不同類別的比較 | 各機台產量比較、各班次良率比較 |
| **柏拉圖（Pareto Chart）** | 頻率由高到低的排名（兼顯累計比例） | 不良原因分析（80/20 法則） |
| **散布圖（Scatter Plot）** | 兩個變數的相關性 | 製程溫度與良率的關係 |
| **管制圖（Control Chart）** | 製程數據與管制界限 | SPC 品質管制 |
| **圓餅圖（Pie Chart）** | 各部分佔整體的比例 | 不良類型佔比（謹慎使用） |

### 2. 工業應用：產量趨勢圖

一張良好的產量趨勢圖應該包含：

- **X 軸**：時間（日期、週次、月份）
- **Y 軸**：產量或良率數值
- **目標線**：清楚標示目標值（如良率目標 95%）
- **異常標記**：明顯標示低於目標的時間點
- **趨勢線**：顯示整體走向（上升、下降或持平）

這樣的圖表讓管理者在幾秒鐘內就能掌握本月生產狀況，遠比閱讀一張充滿數字的 Excel 表格更有效率。

> **工具建議**：對於初學者，Excel 的圖表功能已經足以製作上述大多數圖表。若希望更進一步，Python 的 matplotlib 和 seaborn 套件，或商業 BI 工具（如 Power BI、Tableau）能製作更靈活、更互動的視覺化報表。

## 二十二、學習目標總結

完成本課程後，你應該具備以下能力：

**運算思維與問題拆解**

能夠面對一個複雜的工業問題（如「如何讓報表自動化」、「如何找出生產異常的原因」），運用問題分解、模式辨識、抽象化、演算法思考，將其轉化為可執行的解決步驟。

**基本程式邏輯理解**

能夠閱讀並理解基本程式邏輯（變數、條件判斷、迴圈、函數），即使不限定特定語言，也能表達清楚的程式設計思路。在與 IT 人員溝通系統需求時，能夠精確描述「程式應該如何判斷」、「資料應該如何流動」。

**資料結構的直覺理解**

理解列表、堆疊、佇列、映射、樹等常見資料結構各自的特性與適用情境，在面對不同的工業資料管理問題時，能夠選擇合適的組織方式。

**效率概念的應用**

能夠用直覺理解 O(1)、O(n)、O(n²) 的差異，在評估工業資訊系統的效能問題時，理解「資料量增加對系統速度的影響」，並以此作為系統設計或採購評估的依據之一。

**程式設計與工業流程的連結**

能夠將程式設計的基本概念（輸入→處理→輸出、函數與模組化、資料結構選擇）與工廠的實際流程（生產記錄、品質管制、排程、庫存管理）連結，理解 MES、ERP 等系統背後的資料處理邏輯，並能提出合理的自動化改善方向。
